A Short Introduction

Welcome to the AudioIO docs page! The documentation below is not comprehensive, but it completely covers the user API, as well as certain other elements of the software that may help to illuminate the underlying design patterns upon which it is built. If you are looking for a thorough treatment of all internal elements, inline documentation is provided in the form of docstrings.

File Structure

Here's a fuzzy picture of what the package looks like:

[ROOT]/
  |
  |- README.md                      -->  The README file for GitHub
  |--> audioIO/                     -->  Where the source files live
  |       |
  |       |--> io/                  -->  Holds modules that deal with I/O
  |       |     |
  |       |     |- audioIO.py       -->  Abstract I/O base classes
  |       |     |- wavIO.py         -->  Concrete I/O classes for .WAV files
  |       |     
  |       |
  |       |--> plugins/             -->  Holds modules that deal with plugins
  |               |
  |               |- plugin.py      -->  Plugin class
  |               
  |
  |--> testAudioIO/                 -->  Where the test automation lives
            |
     [lots of stuff]

Some things to notice:

  • At the highest level, the package is split into the audioIO/ directory, which holds the main package contents, and the testAudioIO/ directory, which holds all test automation infrastructure.

  • Within the audioIO/ directory, the source files are split based on whether they deal with I/O (the 'io/' dir) or the core plugin framework (the 'plugin/' dir).

  • The audioIO.py module in the io/ dir holds abstract base classes that define a common interface for audio file I/O in the context of this framework. This interface is then implemented separately (in child classes) for each audio file format that is supported. For example, the wavIO.py module holds concrete classes that implement the interface for .WAV files. See the design overview for more information on this and other design patterns that are exploited.

  • The plugin.py module in the plugin/ dir defines the Plugin class, which is what ties together and coordinates the functionality of the I/O classes.
